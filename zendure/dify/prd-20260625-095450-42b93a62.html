P.2 背景

背景与目标

前后逻辑：此模块作为整个产品的业务基础，定义痛点与目标，后续所有功能均基于这些痛点展开

品牌营销中心在私域社群中处理大量VOC咨询与售后问题，人工压力高，负面舆情易扩散，需要AI助手实现自动答疑、售后工单识别与生成、严重VOC分级告警，并确保高风险场景安全分流。

核心痛点

P0

群组消息量大导致重复答疑成本高，负面情绪未及时安抚易引发品牌风险，售后意图缺乏自动化识别，严重VOC缺少分级告警，以及多客服重复处理同一案件引发用户二次愤怒。

  • 高频Q&A问题反复人工回复
  • 负面反馈响应不及时导致舆情升级
  • 售后诉求依靠人工判断是否建工单
  • 电池召回、品牌声誉等高风险场景无自动分流
  • 仅靠关键词判断导致误报与误回复
  • 多客服重复处理同一案件

目标

P1

建立AI群组答疑助手，实现自动化答疑、售后工单生成、VOC严重等级识别与飞书告警、高风险安全分流及可审计的运营闭环。

  • 基于Q&A知识库和上下文自动回复或安抚
  • 识别售后意图并自动创建CRM工单或引导用户
  • 根据严重程度进行飞书分级告警
  • 对电池召回、供应商低价、品牌声誉等场景AI不介入
  • 沉淀回复记录、判断结果与人工接管数据

目标用户与角色

前后逻辑：定义产品的目标用户及其核心诉求，为后续功能模块设计提供角色依据

产品主要面向社媒运营、社群运营、客服售后团队、品牌营销管理者以及数据中台/AI团队，分别有风险识别、自动答疑、工单结构化、严重告警统计和服务稳定性的诉求。

P.5 核心能力

核心生成流水线

前后逻辑：核心生成流水线为产品的主要功能模块，串联消息接入、意图识别、回复生成、工单处理和告警等环节，后续子模块按顺序展开

以标准化VOC事件为起点，经意图识别与严重等级判断，根据场景执行自动化回复、CRM工单生成或飞书分级告警，并支持人工审核与反馈闭环，实现端到端社群答疑与风险管控。

F1：群组消息接入与VOC解析

从VOC数据中台接入群组消息并完成去重、清洗、语言识别和上下文聚合，生成标准化VOCEvent对象，为后续判断提供基础。

  • 数据输入：群组消息ID、群组信息、用户信息、消息正文、上下文、历史标签等
  • 用户输入：监听群组范围配置、自动/半自动/只告警策略、大促期间高敏感策略
  • 系统输出：标准化VOCEvent、语言识别结果、消息主题分类、情绪倾向、初步严重等级
F2：用户意图识别与场景分类

基于VOCEvent结合群组上下文、历史工单和知识库进行意图分类，输出置信度与是否需要自动回复、工单、告警及禁止AI介入的决策。

  • 建议意图分类：售前咨询、售后咨询、工单跟进、负面反馈、品牌风险、价格敏感、法务极端表达、老板回复、正向互动
  • 关键原则：不以关键词作为单一判断标准，结合上下文与个案描述，高风险场景优先告警而非自动回复，避免让用户重复描述诉求
  • 运营可配置分类规则、高风险关键词组合规则，并人工标注反馈以优化模型
F3：自动化回复生成

根据用户消息、上下文、Q&A知识库及品牌话术生成回复，支持多种发送模式（自动/审核后/仅推荐），并依据安全规则拦截不应回复的场景。

  • 回复策略：正向互动自动感谢，高频售前基于知识库回答，一般售后安抚加引导工单，长期未处理工单提醒客服，AI两轮未解决转人工
  • 安全拦截：电池召回、低价供应商、品牌声誉、B/C端价格问题、老板回复时AI不回复
  • 负向安抚示例模板可配置，支持多语言与地区切换
F4：售后工单意图识别与CRM工单生成

识别用户售后诉求并生成CRM工单草稿，自动创建或引导用户补充信息，同时进行工单去重与案件合并，避免重复建单或多次询问。

  • 工单创建原则：避免重复创建，长期不回复需提醒客服，同一案件多客服处理需合并，尽量减少补充字段由上下文提取
  • 建议CRM工单字段：来源渠道、群组ID、消息ID、用户ID、联系方式、产品型号、订单号、问题类型、摘要、情绪、严重等级、分配团队、状态等
F5：VOC严重等级识别与飞书分级告警

对VOC事件进行严重等级评估，按S0～S4分级，并通过飞书进行实时告警，支持@负责人和创建待办，高风险事件AI不介入。

  • 严重等级：S0最高风险（老板回复、电池召回等）立即告警，S1高风险（价格争议）不介入或内部建议，S2中风险（退款、工单未处理）安抚加告警，S3低风险可不告警，S4正向互动不告警
  • 用户配置：告警规则、飞书群、接收人、SLA
F6：人工审核与反馈闭环

提供人工审核界面，支持确认/驳回/修改回复、调整意图标签和严重等级，沉淀审核结果用于规则优化和评估样本积累。

  • 运营可标记误判并加入规则库或案例库
  • 系统更新VOC数据中台标签和案例库，生成评估数据集，支持后续Playbook规则优化

P.6 流程

用户流程

前后逻辑：以用户视角展示从社群消息进入到最终动作决策与结果回写的全流程，支撑后续架构与非功能需求

社群消息经VOC数据中台接入后，标准化为VOCEvent，Agent依次进行意图识别、情绪分析、严重等级判断及AI介入决策，最后执行自动回复、工单创建或飞书告警，并将结果回写至数据中台以供审计与优化。

流程步骤

P0

从消息接收到最终执行与反馈的端到端用户流程。

  • 社群消息进入VOC数据中台
  • 系统接收/拉取新VOC消息，进行清洗、去重、语言识别与上下文聚合
  • 生成标准化VOCEvent
  • Agent判断流程：意图识别、情绪识别、严重等级识别、是否允许AI介入、是否需要工单/告警/人工审核
  • 动作决策：可自动回复则生成并发送或进入审核池；售后意图明确则生成工单草稿并创建工单；高风险VOC则不发回复仅飞书告警；低置信度或规则冲突转人工审核；正向互动生成互动回复
  • 结果写回：回复记录、工单结果、飞书告警结果写回VOC数据中台，人工反馈进入评估与规则优化
P.7 非功能

非功能需求

前后逻辑：在业务流程确定后，定义实时性、准确性、可靠性、安全性、可审计性等非功能需求，作为架构和技术选型的约束

覆盖实时性（S0/S1告警秒级到1分钟内）、准确性（意图识别与工单判定 >90%）、可靠性（工单生成 >99%且失败重试）、安全性（高风险不自动回复）、可审计性、可配置性、可扩展性、可观测性和合规性等维度。

关键非功能要求

P0

系统在实时性、准确性和安全维度上的核心承诺。

  • 严重VOC告警SLA：S0/S1建议秒级到1分钟，S2可分钟级
  • 意图识别准确率>90%，售后工单意图判定准确率>90%
  • 工单生成成功率>99%，失败需重试与告警
  • 高风险、价格敏感、品牌声誉场景默认不自动回复
  • 每次判断需记录输入、模型输出、规则命中、动作结果与人工反馈
  • 群组策略、告警等级、回复模板、禁用场景支持配置化
  • 支持后续多渠道、多语言、多CRM字段扩展
  • 支持Trace、日志、指标与失败回放的可观测性
  • 用户隐私与订单信息需脱敏存储和最小化使用

P.9 技术

技术选型与架构

前后逻辑：在了解缺口后提出技术选型与总体架构，作为后续开发的基础

建议采用Agent编排+RAG+规则引擎+动作Adapter的工程路线，通过LangGraph或轻量状态机实现决策分支，以LiteLLM网关统一模型调用，结合RAG知识库与配置化规则引擎，并通过CRM Adapter和Feishu Adapter Service隔离外部系统。

总体架构组件

P0

从VOC事件接入到最终动作执行的分层架构组件。

  • VOC Event Ingestion Service：消息接入、去重、清洗、上下文聚合
  • Community Q&A Agent Service：意图识别、严重等级判断、回复生成、工单决策、告警决策
  • RAG Knowledge Service：包含Q&A知识库、VOC分级规则、品牌话术库、历史案例库
  • LiteLLM Gateway：模型路由、成本控制、调用日志、多模型fallback
  • CRM Adapter：工单创建、查询、字段映射
  • Feishu Adapter Service：飞书告警、待办、审批、回写

决策策略：四层决策

P0

推荐采用硬规则拦截 -> LLM意图识别 -> RAG回复生成 -> 动作执行的四层策略，确保高风险场景由确定性规则兜底。

  • 第一层硬规则：老板回复、电池召回、价格争议、品牌声誉等直接禁止自动回复
  • 第二层LLM意图识别：判断咨询、投诉、售后、工单跟进、正向互动等意图
  • 第三层RAG回复生成：仅在允许AI介入时从知识库和话术库检索生成回复
  • 第四层动作执行：根据策略执行自动回复、工单创建、飞书告警或人工审核

Feishu Adapter Service职能

该服务应统一封装飞书侧动作，包括告警卡片发送、@负责人、创建待办、处理状态回写、人工审核卡片、失败重试、API调用日志和权限Token管理，业务Agent只需输出标准动作JSON。

  • 告警卡片发送与分级@负责人
  • 创建待办与处理状态回写
  • 人工审核卡片交互
  • 飞书消息发送失败重试与日志记录
  • 权限和token管理
P.10 里程碑

里程碑与交付节奏

前后逻辑：根据依赖缺口和技术方案规划分阶段交付的里程碑，明确每阶段目标和时间

规划从需求澄清、615前一期MVP（意图识别与告警，默认人工审核）、大促灰度（仅低风险半自动）、CRM工单接入、自动回复灰度到评估与自优化的分期交付路径。

消息接入与异步任务设计

社群消息通过 VOC 数据中台实时推送（Webhook 或消息队列）进入系统，经去重、清洗后生成标准化 VOCEvent，再分发至 Agent 处理。异步任务需保证消息不丢失、不重复消费。

  • 建议采用 Kafka 或云原生消息队列，按群组 ID 分区保证顺序
  • 消费者需实现幂等：基于消息 ID 去重，已处理消息直接跳过
  • 高风险场景（S0/S1）消息优先级最高，独立队列避免被普通消息阻塞
  • 失败消息进入死信队列，支持人工重放或自动重试（指数退避）
  • 监控消息积压和消费延迟，设置告警阈值
权限与安全设计要点

系统涉及社群消息、用户订单、CRM 工单等敏感数据，需从网络隔离、数据脱敏、访问控制和审计日志四个层面保障安全。

  • 网络隔离：Agent 服务仅通过内部 API 与 Adapter 通信，不直接暴露公网
  • 数据脱敏：发送给 LLM 的文本需移除手机号、订单号等 PII；日志中同样脱敏
  • 访问控制：运营人员只能查看所负责群组的回复草稿和告警；客服只能查看相关工单
  • 审计日志：所有人工审核操作（确认/驳回/修改）记录操作人和时间，不可篡改
  • 定期演练：模拟高风险消息误回复场景，验证硬规则拦截和人工接管流程

数据流关键路径与降级策略

定义从消息进入到最终动作的主路径和异常降级路径，确保系统在部分组件不可用时仍能安全运行。
