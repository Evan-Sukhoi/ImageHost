P.1 产品背景

背景与成功指标

前后逻辑：该模块为整体产品背景与目标，是所有后续功能的基础。

品牌营销中心在Facebook私域群组中处理大量VOC（客户声音），当前依赖人工，存在响应压力大、负面情绪安抚不及时、售后意图识别缺失、高风险告警不足等问题。产品目标是构建AI群组答疑助手，实现自动答疑、售后工单生成、严重VOC分级飞书告警、高风险场景安全分流，同时保证可审计与可运营。

P.3 消息接入

F1：群组消息接入与VOC解析

前后逻辑：群组消息接入与VOC解析是整个Agent链路的起点，为后续意图识别与回复生成提供标准化事件。

从VOC数据中台接入社群消息，完成去重、清洗、语言识别、上下文聚合，生成标准化VOC事件对象，并判断是否进入Agent判断流程。支持社媒运营配置监听群组范围、自动回复策略及大促高敏感策略。

数据输入

来自VOC数据中台的社群消息数据，包括消息ID、群组信息、用户信息、消息正文、上下文等。

  • 群组消息ID、群组ID/名称
  • 用户ID/昵称/角色
  • 消息正文、消息语言、消息时间
  • 回复链路/上下文消息
  • 历史VOC标签、是否已处理、是否已生成工单、是否有告警

系统输出

标准化VOC事件对象及预处理结果。

  • 标准化VOCEvent对象
  • 消息语言识别结果
  • 消息主题分类、用户情绪倾向
  • 初步严重等级、是否需要进入Agent判断流程

处理逻辑

P0

消息接入、清洗、聚合，生成标准化事件。

  • 从VOC数据中台拉取或接收群组消息
  • 消息去重、清洗、语言识别、上下文补全
  • 根据群组、用户、时间窗口聚合同一用户连续消息
  • 生成标准化VOC事件，进入后续意图识别流程
用户输入

运营可配置监听范围、回复策略及大促高敏感策略。

  • 配置需要监听的群组范围
  • 配置自动回复/半自动审核/只告警的群组策略
  • 配置特殊活动周期（如615-715营销大促）高敏感策略
P.4 意图识别

F2：用户意图识别与场景分类

前后逻辑：在获得标准化VOC事件后，需要意图识别与场景分类，以决定后续回复、工单或告警动作。

基于VOC事件、上下文、历史记录、Q&A知识库及分级规则，识别售前咨询、售后诉求、工单跟进、负面反馈、品牌风险、价格敏感、法务/极端表达、老板回复、正向互动等意图，并给出是否自动回复、是否需人工审核、是否创建工单、是否飞书告警的判断。

系统输出

P0

意图分类结果及后续动作建议。

  • 意图分类结果及置信度
  • 是否自动回复、是否需要人工审核
  • 是否需要创建工单、是否需要飞书告警
  • 是否禁止AI介入

核心原则

判断需结合关键词、个案描述与上下文，避免单点误判。

  • 不以关键词作为单一判断标准
  • 结合关键词+具体个案描述+上下文判断
  • 高风险、价格敏感、品牌声誉类问题优先告警而非自动回复
  • 用户已表达过的诉求不应让用户重复描述

建议意图分类与默认动作

P.5 回复生成

F3：自动化回复生成

前后逻辑：允许AI介入时，根据意图与知识库生成回复文案。

在允许AI回复的场景下，结合用户消息、上下文、Q&A知识库、品牌话术规范及安抚模板，生成建议回复文本，支持自动发送、待审核草稿或仅推荐三种模式。回复策略按场景严格区分，高风险、价格敏感、老板回复等场景禁止生成或发送回复。

回复策略

P0

不同场景采用不同回复方式。

  • 正向互动：友好互动、感谢支持
  • 高频售前问题：基于Q&A知识库直接回答
  • 一般售后问题：安抚+引导工单
  • 已有工单长期未回复：安抚+提醒客服加快处理
  • AI客服两轮未解决：转人工/提醒客服
  • 电池召回、低价供应商、品牌声誉：AI不回复，飞书告警
  • B/C端价格问题：AI不回复，飞书提醒
  • 老板回复：AI不回复，最高级告警

系统输出与动作

回复文案及执行策略。

  • 建议回复文本、回复语言、引用知识库来源
  • 回复置信度、是否建议自动发送、是否需要人工审核
  • 不回复原因（若被安全规则拦截）
  • 向群组发送自动回复（通过Adapter）
  • 生成待审核回复任务、回复记录写回VOC数据中台
用户输入

运营可选择回复模式和语气。

  • 选择回复模式：自动发送/生成草稿人工确认后发送/仅推荐不发送
  • 配置不同群组的回复语气：正式、友好、简短、安抚优先
P.6 工单集成

F4：售后工单意图识别与CRM工单生成

前后逻辑：当意图识别为售后时，需判断是否生成CRM工单，并完成创建或信息补充。

识别用户售后诉求后，自动判断是否创建CRM工单，生成工单草稿，检测缺失字段并引导补充，避免重复建单。通过CRM Adapter调用API创建或查询工单，确保信息完整性并提醒客服处理。同时发送飞书提醒，支持人工确认。

工单创建注意事项

P0

保证工单创建的准确性与体验。

  • 售后建立工单后长期不回复，应提醒客服加快处理
  • 同一案件多客服处理会引起客户恼怒，需工单去重和案件合并
  • 不应让用户重复描述已表达的诉求
  • 工单字段不完整时，优先提取上下文已有信息，只请求最少补充字段

建议CRM工单字段

CRM系统需映射的核心字段。

  • source_channel, group_id, message_id
  • user_id, user_name, user_contact
  • product_model, order_id
  • issue_type, issue_summary, original_message
  • sentiment, severity, qa_context, assigned_team, status
P.7 分级告警

F5：VOC严重等级识别与飞书分级告警

前后逻辑：基于意图与情绪识别结果，对严重VOC进行分级告警，保证安全分流。

根据严重等级规则表、意图识别及情绪识别结果，将VOC划分为S0~S4五个等级，触发不同的飞书告警动作（立即告警@负责人、提醒客服、仅记录等）。S0/S1高风险场景禁止AI自动回复，确保品牌声誉和召回等事件及时转人工。

动作输出

P0

通过Feishu Adapter Service执行告警与待办。

  • 发送飞书告警卡片（带摘要、上下文、推荐处理人）
  • 创建飞书待办/工单提醒，根据等级@对应负责人
  • 写入告警记录与处理状态

建议严重等级

等级场景AI行为飞书动作
S0 最高风险Bryan回复、电池召回、品牌声誉、价格敏感、工单长期不回复、2400AC熔断不介入立即告警，@社媒负责人和相关负责人
S1 高风险供应商低价讨论、B/C端价格争议、集体投诉苗头不介入或仅生成内部建议分级告警，要求运营处理
S2 中风险退款、发货时效、故障、AI客服回复不当、工单长期未处理安抚+工单/客服提醒飞书提醒客服或运营
S3 低风险一般咨询、轻微不满自动回复或人工审核可不告警，仅记录
S4 正向互动表扬、晒单、普通互动自动互动不告警
P.8 人工闭环

F6：人工审核与反馈闭环

前后逻辑：人工审核与反馈闭环用于修正Agent判断，沉淀案例与规则优化。

运营人员可对Agent生成的回复、意图标签、严重等级进行确认、驳回或修改，标记误判并补充案例库。系统根据反馈更新VOC数据标签和案例库，生成可用于评估的数据集，支持后续规则优化与模型评估，实现持续迭代。

人工输入

P1

审核操作类型。

  • 人工确认/驳回/修改回复
  • 修改意图标签、严重等级
  • 标记是否误判、是否应加入规则库或案例库

系统输出与动作

闭环结果。

  • 审核结果、修正后的意图标签与回复内容
  • 训练/评估样本、规则优化建议
  • 更新VOC数据中台标签
  • 更新案例库，生成可评估数据集
P.9 I/O边界

整体流程与输入输出边界

前后逻辑：整体梳理数据输入、用户输入、系统输出与动作输出的边界，明晰各系统交互点。

社群消息进入VOC数据中台后，经消息接入与解析、意图识别、严重等级判别，进入动作决策（自动回复、工单创建、飞书告警或人工审核），最终结果全部写回数据中台。流程串联了VOC数据中台、Q&A知识库、CRM系统、飞书等多个上下游。

数据输入来源

所需上游数据与依赖状态。

  • VOC数据中台：社群消息、历史标签、群组上下文（已有基础设施）
  • Q&A知识库：售前+售后问答、品牌标准回复（前置依赖暂缺）
  • VOC严重程度分级表：严重等级、处理方式、示例（前置依赖暂缺）
  • CRM系统：工单创建、查询、字段映射（权限与字段映射暂缺）
  • 飞书：告警群、通知、待办、审批（Feishu Adapter Service待开发）
  • 活动日历：615-715营销大促等（需业务维护）

动作输出

Agent触发的系统动作。

  • 自动回复：向群组发送回复（通过Adapter）
  • 创建CRM工单：调用CRM API创建工单
  • 查询CRM工单：避免重复建单，识别长期未处理
  • 发送飞书告警：分级告警卡片
  • 创建飞书待办：为运营或客服创建处理任务
  • 写回数据中台：保存判断、回复、告警、工单、人工反馈
  • 人工接管：高风险或低置信度场景转人工处理
P.10 技术架构

技术架构与决策策略

前后逻辑：确定技术组件与四层决策策略，支撑业务链路可靠执行。

采用Agent编排+RAG+规则引擎+动作Adapter的工程路线。总体架构从VOC Event Ingestion到Community Q&A Agent Service，再通过LiteLLM网关调用LLM，结合RAG知识服务、CRM Adapter和Feishu Adapter Service。决策分为硬规则拦截、LLM意图识别、RAG回复生成、动作执行四层，高风险场景规则优先。

组件选型

核心模块建议选型。

  • Agent编排：LangGraph或轻量自研状态机，明确分支
  • LLM服务：定制化LiteLLM网关，统一路由、日志、成本控制
  • 知识库检索：RAG+向量库+结构化规则表
  • 规则引擎：配置化YAML/JSON+代码规则
  • 飞书集成：Feishu Adapter Service统一封装
  • CRM集成：CRM Adapter，隔离字段、权限、幂等
  • 日志与观测：Langfuse/OpenTelemetry/自研Trace
  • 评估体系：离线标注集+线上人工反馈

四层决策策略

从规则拦截到动作执行的递进决策。

  • 第一层：硬规则拦截（老板回复、电池召回、价格争议等直接禁止自动回复）
  • 第二层：LLM意图识别（判断意图、情感、严重等级、是否允许AI回复）
  • 第三层：策略引擎裁决（根据置信度、群组策略、风险等级决定动作）
  • 第四层：动作执行（自动回复、创建工单、飞书告警或人工审核，并写回数据中台）

P.12 风险

风险与注意事项

前后逻辑：总结关键风险与应对措施，确保交付可靠。

产品涉及品牌声誉、售后承诺等高风险领域，需重点关注误回复、工单重复创建、用户重复描述诉求、告警过多、关键词误判、实时通道不稳定、CRM权限不完整等问题，并分别制定了硬规则拦截、去重键、上下文抽取、分级告警等应对措施。

风险与应对

风险说明应对
误回复高风险问题品牌声誉、价格争议、召回等问题不能让AI直接答硬规则拦截+人工审核
