P.2 背景

背景与目标

前后逻辑：业务背景与项目目标，为整体需求提供基础

品牌营销中心在 Facebook 私域群组中面临大量用户 VOC 处理压力，包括咨询、售后、负面反馈等，人工响应存在效率低、风险高的问题。本项目旨在构建 AI 群组答疑助手，实现自动答疑、售后工单生成、严重 VOC 告警、高风险场景安全分流，并达到意图识别准确率>90%，工单生成成功率>99%等目标。

成功指标

指标目标

售后工单意图判定准确率> 90%
工单生成成功率> 99%
严重 VOC 告警漏报率 (S0)< 1%
AI 不应介入场景误回复率< 1%
运营人工审核通过率> 85%

P.4 核心能力

核心功能

前后逻辑：承接背景与用户需求，定义核心功能模块，按数据流顺序排列

群组消息接入与标准化、意图识别与场景分类、自动回复生成、售后工单意图识别与创建、VOC 严重等级告警、人工审核反馈闭环，六个核心能力按数据流串联。

群组消息接入与 VOC 解析

从 VOC 数据中台拉取社群消息，经过去重、清洗、语言识别、上下文聚合，生成标准 VOCEvent 对象，为后续意图识别提供输入。

用户意图识别与场景分类

基于用户消息、上下文、历史记录和知识库，输出意图分类（售前/售后/负面反馈/风险等）、置信度及是否需要自动回复、工单或告警。

自动化回复生成

根据用户消息、Q&A 知识库、品牌话术生成回复文本，可配置自动发送、人工确认或仅推荐模式，并支持针对不同场景的回复策略。

售后工单意图识别与 CRM 工单生成

识别售后诉求，自动提取信息生成工单草稿，调用 CRM 创建工单，处理去重、字段补全和工单提醒。

VOC 严重等级识别与飞书分级告警

基于规则和上下文判断 VOC 严重等级（S0-S4），对高风险场景不自动回复，仅通过飞书告警通知运营人员。

人工审核与反馈闭环

提供人工确认、驳回、修改回复和标签的能力，沉淀反馈结果用于评估模型和优化规则。

P.6 流程

用户流程

前后逻辑：描述从消息进入系统到最终写回的完整用户流程，串联各功能模块

消息从群组进入系统经过清洗和聚合后，Agent 依次进行意图识别、严重等级判断、动作决策，最后将回复、工单、告警结果写回数据中台。

消息接入与标准化

消息预处理阶段

  • 系统接收/拉取新 VOC 消息
  • 清洗、去重、语言识别、上下文聚合
  • 生成标准化 VOCEvent

Agent 判断

意图与风险评估

  • 意图识别：售前/售后/负面反馈/品牌风险等
  • 情绪识别：正向/中性/负向/强负向
  • 严重等级识别：S0-S4
  • 判断是否允许 AI 介入及是否需要工单/告警

动作决策

根据判断结果执行动作

  • 可自动回复 → 生成回复 → 发送或进入审核
  • 售后意图明确 → 生成 CRM 工单草稿 → 创建工单
  • 高风险 VOC → 不自动回复 → 飞书分级告警
  • 低置信度/规则冲突 → 人工审核
  • 正向互动 → 生成互动回复

结果写回

记录与闭环

  • 回复记录写回 VOC 数据中台
  • 工单结果写回 VOC 数据中台
  • 飞书告警结果写回
  • 人工反馈进入评估与规则优化

核心技术选型推荐

技术领域推荐方案适用原因注意事项

技术选型需重点考虑与现有基础设施的集成方式，避免重复建设。

安全与合规落地建议

围绕 PRD 中高风险场景不自动回复、数据脱敏等要求，给出技术侧加固措施。

  • 在 Agent 决策层增加硬规则拦截器，对 S0/S1 场景强制阻断自动回复，仅允许生成内部建议

开源调研方向

前后逻辑：针对社群答疑 Agent 的核心能力（意图识别、RAG、工单生成、告警），提供开源调研方向，帮助团队评估可复用组件，避免从零开发

社群群组答疑助手涉及 NLP/LLM 应用、知识库检索、工单自动化、告警通知等通用能力，开源社区已有部分可参考方案。为避免重复造轮子并加速落地，本补充给出三个重点调研方向，每个方向提供检索关键词、评估标准、可替代类别及选型风险，供技术选型时参考。

Agent 编排与对话管理框架

建议

调研可复用的 LLM Agent 编排框架，用于意图路由、多步推理和工具调用。

  • 检索关键词：LLM agent framework, conversational AI orchestration, intent routing, tool calling
  • 评估标准：是否支持自定义工具接入、多轮对话状态管理、流式输出、模型无关性、社区活跃度、许可证
  • 可替代类别：LangChain/LangGraph、Semantic Kernel、Haystack、自研轻量 Agent 引擎
  • 选型风险：框架过度抽象可能导致调试困难；部分框架对非 OpenAI 模型支持有限；需评估延迟与 token 消耗

RAG 知识库与检索增强方案

建议

调研开源的 RAG 流水线组件，用于构建 Q&A 知识库的语义检索与答案生成。

  • 检索关键词：RAG pipeline, vector database, embedding retrieval, question answering system
  • 评估标准：检索准确率、多语言支持、知识库更新机制、与 LLM 的集成方式、部署复杂度
  • 可替代类别：LangChain RAG、LlamaIndex、Haystack RAG、自研检索模块
  • 选型风险：向量数据库选型需考虑运维成本；知识库 chunk 策略直接影响召回质量；需建立评估数据集持续优化

消息接入与工单/告警适配器

建议

调研消息队列、CRM 连接器、飞书通知等可复用组件，减少自研 Adapter 工作量。

  • 检索关键词：message queue adapter, CRM integration open source, Feishu bot SDK, alerting pipeline
  • 评估标准：是否支持所需消息中间件、CRM API 封装程度、飞书卡片消息模板化、错误重试与监控
  • 可替代类别：Apache Camel、Temporal、Prefect、自研 Adapter 服务
  • 选型风险：开源 Adapter 可能不兼容内部 CRM 字段映射；飞书 SDK 版本更新频繁；需关注社区维护状态
