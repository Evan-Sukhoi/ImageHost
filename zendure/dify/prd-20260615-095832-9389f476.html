P.1 项目概述

背景、目标与用户

前后逻辑：项目定义与业务背景，为后续所有模块提供目标、用户和成功指标的依据

品牌营销中心在Facebook私域群组中需要持续处理用户咨询、负面反馈、售后诉求与价格争议，当前人工答疑压力大。本项目构建AI群组答疑助手，实现自动化群组答疑、售后意图识别与工单生成、VOC严重等级识别与飞书分级告警，并将高风险场景安全分流。目标用户覆盖社媒运营、社群运营、客服、品牌管理者和技术团队。

意图识别准确率>90%
工单判定准确率>90%
工单生成成功率>99%
高风险误回复率<1%
运营审核通过率>85%

业务痛点

背景

私域群组消息量大，人工响应压力高，负面情绪需及时安抚，售后意图识别依赖人工，严重VOC需分级告警，多客服重复处理引发用户不满。

  • 高频QA人工重复回复
  • 退款/发货/客服不满响应不及时
  • 售后工单创建缺乏自动化
  • 电池召回等高风险场景需立即告警
  • 同一案件多客服重复询问

核心目标

目标

构建面向私域社群的AI答疑助手，实现四个核心能力。

  • 自动化群组答疑与安抚回复
  • 售后意图识别与CRM工单生成
  • VOC严重等级识别与飞书分级告警
  • 高风险场景安全分流，AI不介入

目标用户

用户

涉及社媒运营、社群运营、客服、品牌管理者及数据/AI团队。

  • 社媒运营：快速识别风险，减轻重复答疑
  • 社群运营：自动处理高频问题
  • 客服团队：获得结构化工单信息
  • 品牌管理者：严重VOC告警与趋势
  • 数据中台/AI团队：提供稳定服务
P.2 消息接入

群组消息接入与VOC解析

前后逻辑：作为整个Agent流程的起点，将社群消息标准化为VOC事件后，供下游意图识别模块消费

从VOC数据中台接入或拉取社群消息，进行去重、清洗、语言识别和上下文聚合，输出标准化VOC事件对象，为后续意图识别提供结构化输入。支持运营配置监听群组、自动回复策略和大促特殊策略。

数据输入

P0

来自VOC数据中台的社群消息及历史标签。

  • 群组消息ID/群组信息
  • 用户ID/昵称/角色
  • 消息正文/语言/时间
  • 上下文消息与历史VOC标签
  • 附件/图片/链接标记

用户输入与配置

P0

运营配置监听范围与策略。

  • 配置需要监听的群组
  • 启用自动回复/半自动审核/只告警策略
  • 配置活动期间高敏感策略如615-715大促

系统输出

P0

标准化VOCEvent对象。

  • 消息语言识别结果
  • 消息主题分类
  • 用户情绪倾向
  • 初步严重等级
  • 是否进入Agent判断流程

处理逻辑

P0

核心处理步骤。

  • 拉取或接收群组消息
  • 去重、清洗、语言识别、上下文补全
  • 聚合同一用户连续消息
  • 生成标准化VOC事件
P.3 意图识别

用户意图识别与场景分类

前后逻辑：接收标准化VOC事件后，进行意图识别和场景分类，决策后续动作分支（回复/工单/告警/人工审核）

基于VOC事件、群组上下文、历史记录和知识库，识别用户意图（售前/售后/工单跟进/负面反馈/品牌风险等），输出意图分类、置信度、是否自动回复、是否需要创建工单或飞书告警、是否禁止AI介入。采用关键词+上下文+个案描述判断，避免单一关键词误判。

意图分类与默认动作

P0

一级和二级意图及对应默认动作。

  • 售前咨询→自动回复/引导官网
  • 售后咨询→安抚+引导工单/创建工单
  • 工单跟进→安抚+提醒客服
  • 负面反馈→安抚+分级告警
  • 品牌风险→AI不介入+飞书告警
  • 价格敏感→AI不介入+飞书提醒
  • 法务/极端表达→结合上下文预警
  • 老板回复→最高级告警转社媒运营
  • 正向互动→自动互动回复

关键原则

P0

判断准则。

  • 不单一依赖关键词
  • 结合关键词+个案描述+上下文
  • 高风险/价格敏感/品牌声誉优先告警而非自动回复
  • 用户已表达诉求不再重复询问

系统输出

P0

意图分类结果输出。

  • 意图分类与置信度
  • 是否自动回复/人工审核
  • 是否需要工单/飞书告警
  • 是否禁止AI介入
P.4 回复生成

自动化回复生成

前后逻辑：根据意图分类和回复策略生成回复文本，支持自动发送或人工审核后发送

结合用户消息、上下文、Q&A知识库和品牌话术规范，生成建议回复文本。支持运营选择自动发送、生成草稿人工确认后发送，或仅推荐回复。可配置不同群组的回复语气。高风险场景自动禁止回复并输出原因。

回复模式与策略

P0

不同场景的回复方式。

  • 正向互动→友好互动感谢
  • 高频售前→基于Q&A知识库直接回答
  • 一般售后→安抚+引导工单
  • 工单长期未回复→安抚+提醒客服
  • AI两轮未解决→转人工
  • 高风险场景→不回复只告警

安全拦截与不回复

P0

禁止回复场景。

  • 电池召回、低价供应商、品牌声誉
  • B/C端价格问题
  • 老板回复
  • 遇拦截输出不回复原因

负向安抚模板

P1

标准安抚示例，沉淀到品牌话术库。

  • 示例：我们非常遗憾您有负面经历...请将工单号码发送至support@zendure.com
  • 按语言、地区、场景切换
P.5 工单管理

售后工单意图识别与CRM工单生成

前后逻辑：根据售后意图生成工单草稿，调用CRM API创建工单，去重并提醒客服处理

识别用户售后诉求后，生成CRM工单草稿，自动提取上下文已有信息，尽量少补充字段。调用CRM API创建工单，支持去重和案件合并。工单创建后提醒客服，写回工单结果到数据中台。

工单创建流程

P0

主要步骤。

  • 售后意图识别
  • 提取已有上下文信息
  • 生成工单草稿与缺失字段列表
  • 检查是否满足自动创建条件
  • 调用CRM API创建工单
  • 去重与案件合并，避免重复建单
  • 提醒客服处理长期未响应工单

CRM工单字段

P0

建议字段结构。

  • source_channel：来源渠道
  • group_id / message_id
  • user_id / name / contact
  • product_model / order_id
  • issue_type / summary / original_message
  • sentiment / severity / qa_context
  • assigned_team / status
P.6 告警管理

VOC严重等级识别与飞书分级告警

前后逻辑：对严重VOC进行分级告警，通过飞书通知对应人员，S0/S1立即告警，AI不介入

根据VOC事件、意图和情绪，结合严重等级规则，判定S0-S4等级。通过Feishu Adapter Service发送飞书告警卡片，创建待办，@相关负责人。S0/S1场景禁止AI回复，由社媒运营接管。

告警等级与AI行为

P0

分级标准。

  • S0最高风险：老板回复、召回、品牌声誉、价格敏感、工单长期不回复→不介入，立即告警
  • S1高风险：供应商低价、B/C端价格争议、集体投诉→不介入，分级告警
  • S2中风险：退款、发货、故障、重复不满→安抚+工单提醒，飞书提醒
  • S3低风险：一般咨询→自动回复或审核，可不告警
  • S4正向：表扬晒单→自动互动不告警

飞书告警动作

P0

告警输出。

  • 发送飞书告警卡片
  • @指定的社媒负责人/客服
  • 创建飞书待办
  • 写入告警记录与处理状态

人工审核与反馈闭环

前后逻辑：提供人工确认/驳回/修改机制，反馈修正结果并用于规则优化

P.9 技术策略

技术选型与决策策略

前后逻辑：技术选型与Agent决策策略，指导工程实现

采用Agent编排+RAG+规则引擎+动作Adapter的四层架构，不训练专用大模型。明确硬规则拦截、LLM意图识别、RAG回复生成、策略引擎裁决与动作执行的分层。组件选型包括LangGraph、LiteLLM网关、Feishu Adapter Service等。

四层决策架构

架构

决策流程。

  • 第一层：硬规则拦截（老板回复、召回等禁止自动回复）
  • 第二层：LLM意图识别（咨询/投诉/售后/正向等）
  • 第三层：RAG回复生成（仅在允许AI介入时）
  • 第四层：策略引擎裁决+动作执行（自动回复/工单/告警/审核）

关键组件

选择

推荐组件及原因。

  • Agent编排：LangGraph或轻量状态机
  • LLM服务：定制LiteLLM网关（路由/日志/fallback）
  • 知识库检索：RAG+向量库+结构化规则
  • 规则引擎：YAML/JSON配置+代码规则
  • 飞书集成：Feishu Adapter Service统一封装
  • CRM集成：CRM Adapter隔离权限与幂等
  • 日志观测：Langfuse/OpenTelemetry

示例

P.10 里程碑与风险

里程碑与关键风险

前后逻辑：实施里程碑和风险应对，确保交付节奏与风险可控

一期MVP 615前上线，实现消息接入、意图识别、告警和回复草稿，暂不自动回复。大促期间灰度低风险半自动回复。后续接入CRM工单和自动回复灰度。主要风险包括误回复高风险、工单重复、用户重复描述、告警疲劳等，均已制定应对措施。

里程碑

计划

阶段目标。

  • M0：补齐Q&A知识库、VOC分级表等前置依赖
  • M1：615前MVP：接入、意图、告警、回复草稿，全推送人工审核
  • M2：大促期间灰度：低风险半自动回复
  • M3：CRM工单接入：自动生成与去重
  • M4：自动回复灰度启用
  • M5：评估与自优化

关键风险与应对

风险

主要风险。

  • 误回复高风险：硬规则拦截+人工审核
  • 工单重复：CRM查询+去重键
  • 用户重复描述：上下文抽取最小补充字段
  • 告警疲劳：分级告警+聚合摘要
  • 关键词误判：关键词+上下文+个案描述
  • 实时推送不稳定：优先事件推送/Webhook
